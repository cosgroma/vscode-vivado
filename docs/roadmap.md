# Vivado Roadmap

This page describes the to-be Vivado support for `cosgroma/vscode-vivado`.

The current codebase is a Vitis HLS baseline. The roadmap below is the desired product direction for Vivado-centered FPGA development in VS Code.

## Starting Point

The inherited extension is still strongly shaped around Vitis HLS:

- Extension activation currently depends on `hls.app` files.
- Project discovery currently builds `HLSProject` objects.
- The Projects tree currently shows HLS source files, test bench files, and HLS solutions.
- Run commands currently call `vitis-run` for C simulation, C synthesis, and C/RTL cosimulation.

Vivado support should be added as a parallel workflow first. Do not force Vivado concepts into the HLS model. Introduce Vivado-specific project models and task helpers, then unify shared UI behavior once both workflows are clear.

## Layered Feature Plan

### Core Vivado Foundation

Add the basic Vivado surface area before deeper project behavior:

- Activate on `**/*.xpr`.
- Later consider activation on `**/*.bd`, `**/*.xci`, and script-only Vivado workspaces.
- Add settings for the Vivado install path, executable path, environment setup script, project discovery globs, batch-mode defaults, report output paths, and log preservation.
- Add a reusable `vivadoRun(...)` helper that executes `vivado -mode batch -source <script.tcl>` through VS Code tasks.
- Keep generated TCL visible so every extension action can be reproduced in a terminal or CI job.
- Support Windows first while keeping command construction portable.

### Vivado Project Model

Create explicit Vivado domain models instead of reusing HLS classes:

- `VivadoProject`
- `VivadoFileset`
- `VivadoRun`
- `VivadoIp`
- `VivadoBlockDesign`
- `VivadoReport`

Initial metadata should include project name, path, part, board, top module, source filesets, simulation filesets, constraint sets, IP, block designs, and runs.

Prefer Vivado TCL queries as the source of truth for project state. Parsing `.xpr` files can be useful for fast discovery, but Vivado's Tcl API is the safer authority for resolved filesets, generated files, active runs, and IP state.

### Project Tree

The tree should feel like a compact Vivado cockpit rather than an HLS tree with renamed labels:

- Design Sources
- Simulation Sources
- Constraints
- IP Sources
- Block Designs
- Memory Files
- TCL Scripts
- Runs
- Reports
- Hardware

Run nodes should expose synthesis, implementation, bitstream generation, simulation, reset/clean actions, and report links. Report nodes should make timing, utilization, DRC, methodology, and power output easy to open.

### Run Management

Expose common Vivado flows as VS Code commands and tasks:

- Open the project in the Vivado GUI.
- Elaborate RTL.
- Run behavioral simulation.
- Run synthesis.
- Run implementation.
- Generate bitstream.
- Reset or clean selected runs.
- Open generated reports.
- Run checked-in TCL scripts.
- Show the exact TCL generated by the extension.

The first implementation can use project-mode TCL. Script-mode support should follow so users with checked-in build scripts can still get task integration and diagnostics.

### Diagnostics And Reports

Vivado output should become actionable inside VS Code:

- Parse task output into Problems diagnostics.
- Link warnings and errors back to RTL, XDC, Tcl, IP, and block design files when Vivado emits usable file and line information.
- Add problem matchers for common Vivado message families such as `[Synth 8-...]`, `[Place 30-...]`, `[Route 35-...]`, `[Timing 38-...]`, `[DRC ...]`, and `[Vivado 12-...]`.
- Preserve logs for failed, cancelled, and successful runs based on user settings.
- Surface report summaries for worst negative slack, total negative slack, failing clocks, utilization, critical DRCs, and power estimates.
- Provide direct links from run nodes to generated reports.

### Simulation And XSim

Vivado Simulator support should be a first-class workflow:

- Extend behavioral simulation beyond project-mode fileset launches.
- Support `xvhdl`, `xvlog`, `xelab`, and `xsim` where standalone simulator commands are useful.
- Launch waveform viewing in Vivado.
- Save and reopen waveform configurations.
- Generate VCD or SAIF output when requested.
- Support post-synthesis and post-implementation timing simulation later.
- Leave room for AXI transaction-aware helpers because Vivado/XSim can expose AXI Memory-Mapped and AXI4-Stream transaction views.

### IP And Block Designs

IP and IP Integrator support are required for practical Vivado projects:

- Show `.xci` IP cores with status.
- Show stale, locked, or upgradeable IP warnings.
- Run `generate_target`, `synth_ip`, and `upgrade_ip`.
- Show `.bd` block designs.
- Validate block designs.
- Generate block design output products.
- Generate or update HDL wrappers when appropriate.
- Open IP customization and block design views in the Vivado GUI.
- Surface address maps when Vivado exposes them.

### Hardware Programming And Debug

Hardware support should come after project discovery, run management, diagnostics, simulation, and IP are reliable:

- Program a device with a generated `.bit` file.
- Associate `.ltx` debug probes.
- Connect to `hw_server`.
- List hardware targets and devices.
- Open Hardware Manager.
- Program configuration memory.
- Support basic Tcl-backed ILA and VIO actions.
- Consider XVC and remote hardware flows after local hardware flows work.

### HLS And Vivado Together

Vitis HLS support can remain useful for projects that generate Vivado IP:

- Keep inherited HLS project discovery and run commands available.
- Show HLS and Vivado projects together in multi-project workspaces.
- Connect HLS export outputs to Vivado IP repository or IP packaging flows.
- Avoid making either workflow feel like a bolt-on to the other.

## Implementation Order

Build the Vivado workflow in small, testable slices:

1. Rename or generalize user-facing branding and configuration enough that Vivado support can live beside HLS.
2. Add activation for `.xpr` projects.
3. Introduce `VivadoProject` and `VivadoProjectManager` without deleting HLS support.
4. Add `vivadoRun(...)` for Tcl-backed Vivado tasks.
5. Render a minimal Vivado tree with sources, constraints, runs, and reports.
6. Add commands for opening the Vivado GUI, running synthesis, running implementation, and generating a bitstream.
7. Add generated TCL preview plus reset and clean commands for selected runs.
8. Add diagnostics and report parsing.
9. Add XSim simulation commands.
10. Add IP and block design commands.
11. Add hardware programming and debug commands.

## Project Editing

Project modification commands should wait until read-only inspection is reliable:

- Add and remove design sources.
- Add and remove simulation sources.
- Add and remove constraint files.
- Manage filesets.
- Add IP repository paths.
- Generate block design wrappers.
- Update project settings only through explicit user commands.

Project editing should be conservative, test-backed, and explicit about what it changes.

## Design Principles

- Prefer TCL-backed actions over hidden IDE state.
- Treat project files as user-owned source, not disposable generated data.
- Make read-only inspection reliable before adding write commands.
- Keep logs and diagnostics visible.
- Support Windows first, while keeping paths and commands portable where possible.
